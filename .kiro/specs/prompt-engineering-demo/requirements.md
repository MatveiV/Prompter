# Документ требований

## Введение

Скрипт `prompt_engineering_demo.py` — интерактивная демонстрация техник prompt engineering.
Пользователь выбирает бытовую мини-задачу, скрипт формирует два промпта (базовый и улучшенный с few-shot примером),
отправляет оба через Z.AI API, сравнивает ответы и генерирует артефакт README.md с пошаговыми инструкциями.

## Глоссарий

- **Script** — скрипт `prompt_engineering_demo.py`
- **Task** — бытовая мини-задача, выбранная пользователем из предопределённого списка
- **Base_Prompt** — базовый промпт по схеме: роль → контекст → задача → формат ответа
- **Enhanced_Prompt** — улучшенный промпт, дополненный few-shot примером
- **Response** — ответ модели в формате JSON `{ "title": "...", "steps": [...], "notes": [...] }`
- **Comparison** — сравнительный анализ двух ответов по трём критериям
- **Artifact** — файл `README.md`, сгенерированный на основе итогового ответа
- **Provider** — провайдер API (ZAI, ProxyAPI, GenAPI), определённый в `config.py`
- **Model** — языковая модель, выбранная пользователем из списка доступных для провайдера
- **Client** — унифицированный клиент `openai_client.py`, функция `chat()`

## Требования

### Требование 1: Выбор задачи

**User Story:** Как пользователь, я хочу выбрать бытовую мини-задачу из списка, чтобы увидеть, как prompt engineering улучшает ответ модели на конкретном примере.

#### Критерии приёмки

1. THE Script SHALL отображать нумерованный список из не менее 5 бытовых мини-задач при запуске.
2. WHEN пользователь вводит номер задачи, THE Script SHALL загружать соответствующую задачу и переходить к выбору модели.
3. IF пользователь вводит номер, отсутствующий в списке, THEN THE Script SHALL отобразить сообщение об ошибке и повторно запросить ввод.
4. THE Script SHALL включать в список задач: напоминание пить воду, расчёт скидки, приветственный бот, план утренней зарядки, список покупок.

---

### Требование 2: Выбор провайдера и модели

**User Story:** Как пользователь, я хочу выбрать провайдера и модель из доступных, чтобы контролировать, через какой API выполняется запрос.

#### Критерии приёмки

1. THE Script SHALL отображать список провайдеров из `config.py` (ZAI, ProxyAPI, GenAPI).
2. WHEN пользователь выбирает провайдера, THE Script SHALL отображать список моделей этого провайдера с их метками.
3. WHEN пользователь выбирает модель, THE Script SHALL зафиксировать параметры: провайдер, идентификатор модели, temperature=0.2.
4. IF переменная окружения API-ключа выбранного провайдера не задана, THEN THE Script SHALL отобразить сообщение об ошибке и завершить выполнение.
5. THE Script SHALL использовать temperature=0.2 для всех запросов без возможности изменения пользователем.

---

### Требование 3: Формирование промптов

**User Story:** Как разработчик, я хочу, чтобы скрипт автоматически формировал два промпта по заданной схеме, чтобы продемонстрировать разницу между базовым и улучшенным подходами.

#### Критерии приёмки

1. THE Script SHALL формировать Base_Prompt по схеме: роль → контекст → задача → формат ответа для каждой Task.
2. THE Script SHALL формировать Enhanced_Prompt, добавляя к Base_Prompt few-shot пример с корректным JSON-ответом.
3. THE Script SHALL включать в системную инструкцию обоих промптов требование возвращать ответ строго в формате JSON: `{ "title": "...", "steps": ["...", "..."], "notes": ["...", "..."] }`.
4. THE Script SHALL хранить определения всех Task и их промптов в отдельной структуре данных внутри скрипта (не в `prompts.json`).

---

### Требование 4: Выполнение запросов к API

**User Story:** Как пользователь, я хочу, чтобы скрипт последовательно отправлял оба промпта через API, чтобы получить два сопоставимых ответа.

#### Критерии приёмки

1. THE Script SHALL вызывать функцию `chat()` из `openai_client.py` для отправки Base_Prompt и получения первого Response.
2. THE Script SHALL вызывать функцию `chat()` из `openai_client.py` для отправки Enhanced_Prompt и получения второго Response.
3. THE Script SHALL использовать одинаковые параметры (модель, temperature=0.2) для обоих вызовов.
4. IF вызов `chat()` возвращает исключение, THEN THE Script SHALL отобразить текст ошибки и завершить выполнение.
5. THE Script SHALL отображать индикатор прогресса (текстовое сообщение) перед каждым запросом к API.

---

### Требование 5: Валидация JSON-ответов

**User Story:** Как разработчик, я хочу, чтобы скрипт проверял корректность JSON в ответах модели, чтобы сравнение было основано на структурированных данных.

#### Критерии приёмки

1. WHEN Response получен от API, THE Script SHALL попытаться распарсить его как JSON.
2. THE Script SHALL проверять наличие ключей `title` (строка), `steps` (список строк) и `notes` (список строк) в распарсенном JSON.
3. IF Response не является валидным JSON, THEN THE Script SHALL пометить его как «невалидный» и продолжить сравнение с пометкой.
4. IF Response является валидным JSON, но не содержит обязательных ключей, THEN THE Script SHALL пометить его как «неполный» и продолжить сравнение с пометкой.
5. THE Script SHALL извлекать JSON из ответа, даже если модель обернула его в markdown-блок (```json ... ```).

---

### Требование 6: Сравнение ответов

**User Story:** Как пользователь, я хочу видеть сравнение двух ответов по понятным критериям, чтобы оценить эффект от улучшения промпта.

#### Критерии приёмки

1. THE Script SHALL сравнивать два Response по трём критериям: соответствие JSON-формату, полезность steps, лаконичность notes.
2. THE Script SHALL отображать результат сравнения в терминале в виде таблицы или структурированного текста.
3. THE Script SHALL определять соответствие JSON-формату как булево значение (валидный / невалидный) для каждого Response.
4. THE Script SHALL определять полезность steps как количество шагов в списке `steps` для каждого Response.
5. THE Script SHALL определять лаконичность notes как среднюю длину строк в списке `notes` (в символах) для каждого Response.
6. THE Script SHALL указывать, какой из двух ответов лучше по каждому критерию.

---

### Требование 7: Генерация артефакта README.md

**User Story:** Как пользователь, я хочу получить готовый README.md с пошаговыми инструкциями на основе лучшего ответа, чтобы сразу использовать результат.

#### Критерии приёмки

1. WHEN сравнение завершено, THE Script SHALL выбрать итоговый Response (Enhanced_Prompt, если его JSON валиден; иначе Base_Prompt).
2. THE Script SHALL сгенерировать файл `artifact_README.md` в текущей директории на основе итогового Response.
3. THE Script SHALL включать в `artifact_README.md`: заголовок из поля `title`, нумерованный список из поля `steps`, раздел «Примечания» из поля `notes`.
4. THE Script SHALL добавлять в `artifact_README.md` метаданные: название задачи, имя модели, дату и время генерации.
5. IF итоговый Response невалиден, THEN THE Script SHALL записать в `artifact_README.md` сырой текст ответа с пометкой об ошибке парсинга.
6. WHEN файл `artifact_README.md` успешно записан, THE Script SHALL отобразить путь к файлу в терминале.
