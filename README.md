# PromptingAIbot

Набор CLI-скриптов и Telegram-бот для работы с несколькими AI-провайдерами: **Z.AI**, **ProxyAPI**, **GenAPI**.

## Структура проекта

```
.
├── bot.py                      # Telegram-бот (aiogram 3)
├── config.py                   # Провайдеры, модели, цены, токены
├── context_manager.py          # In-memory контекст диалога на пользователя
├── openai_client.py            # Единый OpenAI-совместимый HTTP-клиент
├── ai_direct.py                # Интерактивный CLI с историей и сохранением сессии
├── prompter.py                 # A/B-тест техник промптинга (zero-shot/few-shot/CoT/role)
├── case_prompter.py            # Тестирование JSON-промптов из папки Prompts/
├── zai_direct.py               # CLI — Z.AI (GLM-модели)
├── proxy_api.py                # CLI — ProxyAPI → OpenAI GPT
├── gen_api.py                  # CLI — GenAPI (GPT / Claude / Gemini / DeepSeek)
├── prompts.json                # Системные промпты для ai_direct.py
├── session.json                # Сохранённая сессия ai_direct.py (создаётся автоматически)
├── artifact_YYYYMMDD_HHMMSS.md # A/B-отчёты prompter.py (создаются автоматически)
├── result_YYYYMMDD_HHMMSS.md   # README-артефакты для пользователя (создаются автоматически)
├── Prompts/                    # Системные промпты в JSON для case_prompter.py
│   ├── 01_summary_prompt.json
│   ├── 02_code_structure.json
│   └── 03_task_planning_prompt.json
├── CASE_PROMPTER.md            # Документация к case_prompter.py
├── .env                        # Секреты (не коммитить)
├── .env.example                # Шаблон .env
├── requirements.txt            # Зависимости
└── .gitignore
```

## CLI-скрипты

### ai_direct.py — интерактивный CLI с историей

Основной скрипт для диалога с моделью. Поддерживает выбор провайдера, модели, температуры и системных промптов из `prompts.json`. Сохраняет сессию между запусками.

```bash
python ai_direct.py
```

Шаги при запуске:
1. Если найден `session.json` — предложит продолжить предыдущую сессию
2. Выбор провайдера (Z.AI / ProxyAPI / GenAPI)
3. Выбор модели из списка
4. Настройка температуры
5. Выбор системных промптов из `prompts.json` (можно несколько через запятую, или `0` — без промпта)

Команды в диалоге:

| Команда | Действие |
|---------|----------|
| `/model` | Сменить провайдера, модель и температуру |
| `/new` | Очистить историю (промпты сохраняются) |
| `/exit` | Выйти и сохранить сессию в `session.json` |

### prompter.py — A/B-тест техник промптинга

Тестирует несколько техник промптинга на одной задаче, сравнивает результаты и сохраняет полный отчёт.

```bash
python prompter.py
```

Шаги при запуске:
1. Выбор бытовой задачи из списка (вода, скидка, бот, зарядка, покупки)
2. Выбор провайдера и модели (таблица показывает допустимый диапазон температуры для каждой модели)
3. Настройка параметров: temperature, max_tokens
4. Выбор техник для A/B-теста (можно несколько через запятую или все)
5. Последовательный запуск каждой техники с одинаковыми параметрами
6. Сравнительная таблица в терминале с рангами
7. Сохранение полного отчёта `artifact_YYYYMMDD_HHMMSS.md`

Поддерживаемые техники:

| Техника | Описание |
|---------|----------|
| zero-shot | Прямой запрос без примеров и подсказок |
| few-shot | Запрос с примером корректного ответа |
| chain-of-thought | Запрос с пошаговым рассуждением (CoT) |
| role-based | Запрос с явным указанием роли эксперта |

> Важно: для получения валидного JSON используй temperature=0.2. Значения выше 1.0 приводят к нечитаемым ответам — скрипт предупредит и попросит подтверждение.

Каждый запуск создаёт два файла:
- `artifact_YYYYMMDD_HHMMSS.md` — полный A/B-отчёт: параметры, промпты, ответы, стоимость, сравнение итерации качества
- `result_YYYYMMDD_HHMMSS.md` — итоговый README для пользователя: лучший ответ (шаги + примечания) + сводная таблица A/B-анализа и сравнение итерации zero-shot → few-shot

#### Соответствие заданию

| Пункт | Реализация |
|-------|-----------|
| 1. Выбор задачи | 5 бытовых задач на выбор |
| 2. Схема промпта | Все техники строятся по схеме: роль → контекст → задача → формат |
| 3. Параметры | Фиксируются: модель, temperature, max_tokens, системные инструкции |
| 4. Итерация качества | zero-shot (базовый) → few-shot (+ мини-пример) с теми же параметрами |
| 5. Сравнение | Раздел "Сравнение итерации" в обоих артефактах: JSON-формат, шаги, лаконичность notes |
| 6. Артефакт | `result_YYYYMMDD_HHMMSS.md` — итоговый ответ победителя + полный анализ промптов (таблица A/B + итерация) |

#### Глоссарий параметров

| Параметр | Определение |
|----------|-------------|
| Температура | Степень случайности ответа модели. `0.0` — детерминированный, точный ответ. `2.0` — максимально творческий. Диапазон зависит от модели (обычно `0.0–1.0` для GLM, `0.0–2.0` для GPT). Рекомендуется `0.2` для структурированных задач. Значения выше `1.0` при запросе JSON-вывода приводят к нечитаемым ответам — скрипт предупреждает и запрашивает подтверждение. |
| Токен | Минимальная единица текста, обрабатываемая моделью (~0.75 слова в русском). Стоимость запроса считается по числу токенов на входе (промпт) и выходе (ответ). |
| Контекст | Максимальное число токенов, которое модель может обработать за один запрос (промпт + ответ). Например, `200K` = 200 000 токенов ≈ ~150 000 слов. |
| Max tokens | Ограничение на длину ответа модели в токенах. Не влияет на длину промпта. |
| Статус | Результат валидации JSON-ответа модели: `valid` — корректный JSON с полями `title`, `steps`, `notes`; `incomplete` — JSON есть, но поля неполные; `invalid` — ответ не является JSON. |
| Ранг | Порядковое место техники в A/B-тесте по убыванию итогового балла (1 = лучший). |
| Шаги (steps) | Количество элементов в поле `steps` JSON-ответа. Больше шагов — более детальный ответ. |
| Ср. длина notes | Средняя длина строк в поле `notes` в символах. Меньше — лаконичнее. |
| Стоимость | Стоимость запроса в рублях: `(токены_вход / 1000 × цена_вход) + (токены_выход / 1000 × цена_выход)`. Для бесплатных моделей = 0. |

#### Алгоритм A/B-теста

Все техники запускаются последовательно с **одинаковыми параметрами** (модель, temperature, max_tokens) и одинаковым пользовательским сообщением `"Выполни задачу."`. Различается только системный промпт.

Для каждого ответа вычисляются метрики:

```
json_valid    = (status == "valid")          # булево
steps_count   = len(parsed["steps"])         # целое, 0 если невалиден
avg_notes_len = mean(len(s) for s in notes)  # вещественное, 0.0 если невалиден
```

#### Формула ранга

Итоговый балл техники:

```
score = json_valid × 1.0
      + steps_count × 0.1
      - avg_notes_len × 0.001
```

Техники сортируются по убыванию `score`. Ранг 1 присваивается технике с наибольшим баллом.

Логика формулы:
- `json_valid × 1.0` — главный критерий: валидный JSON даёт +1 балл
- `steps_count × 0.1` — больше шагов = более детальный ответ = лучше
- `avg_notes_len × 0.001` — штраф за многословность: короткие notes лаконичнее

### case_prompter.py — тестирование JSON-промптов

Читает системные промпты из `Prompts/*.json`, принимает вопрос от пользователя и отправляет запрос выбранному провайдеру. Поддерживает повтор одного промпта с разными моделями и параметрами для сравнения.

```bash
python case_prompter.py
```

Шаги при запуске:
1. Выбор промптов из `Prompts/` (один, несколько через запятую, или все)
2. Выбор провайдера и модели
3. Настройка temperature и max_tokens
4. Ввод вопроса (или использование `test_input` из JSON одной клавишей)
5. Ответ выводится в терминал с токенами и стоимостью
6. Результат сохраняется в `cases/case_<id>_<timestamp>.md`
7. Предложение повторить с другими параметрами или моделью

Подробная документация: [CASE_PROMPTER.md](CASE_PROMPTER.md)

### zai_direct.py / proxy_api.py / gen_api.py — одиночные запросы

Простые скрипты для одного запроса к конкретному провайдеру.

```bash
python zai_direct.py   # Z.AI (GLM-модели, есть бесплатные)
python proxy_api.py    # ProxyAPI → OpenAI GPT
python gen_api.py      # GenAPI → GPT / Claude / Gemini / DeepSeek
```

Каждый скрипт интерактивно предлагает:
1. Выбрать модель из списка
2. Ввести system message (опционально)
3. Ввести запрос
4. Задать temperature и max_tokens

## Telegram-бот

```bash
python bot.py
```

### Команды

| Команда | Описание |
|---------|----------|
| `/start` | Приветствие + мастер настройки сессии |
| `/setup` | Выбрать провайдера, модель, temperature, max_tokens |
| `/info` | Текущие настройки сессии |
| `/clear` | Очистить историю диалога |
| `/report` | Отчёт о прогонах моделей |
| `/help` | Справка по командам |

Написать `очистить контекст` — тоже сбрасывает историю.

### Логика бота

- Каждый пользователь имеет свой контекст (последние 20 сообщений, in-memory).
- `/setup` сбрасывает контекст и позволяет сменить провайдера/модель/параметры.
- Ошибка 402 (недостаточно средств) показывается понятным сообщением.

### /report — отчёт о прогонах

```
#<run> <Модель> [<Провайдер>]
  t=<temperature> max=<max_tokens> | <эффект> | <finish_reason>
  токены: <total> (in <prompt> / out <completion>) | <стоимость ₽>
```

Эффект по температуре: `t < 0.4` → `сжато/факт` | `0.4–0.8` → `баланс` | `t > 0.8` → `креатив`

## Архитектура

```
ai_direct.py / prompter.py / bot.py
  ├── config.py          — PROVIDERS dict (ключи, URL, модели, цены)
  ├── context_manager.py — _store: dict[user_id → session]  (только bot.py)
  └── openai_client.py   — openai.OpenAI(base_url=...) → chat()
```

Все три провайдера используют OpenAI-совместимый API — один клиент работает для всех.
`proxy_api.py` и `gen_api.py` используют `requests` напрямую (без SDK).
